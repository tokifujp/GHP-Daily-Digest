# GeminiとClaudeをGASで両方試してプロジェクトダイジェストを自動化した話

**読了時間: 約8分**

## この記事を書いた人・向いている人

自分はインフラ・バックエンドが本職だが、小規模チームではプロジェクト管理も兼務している、いわゆるプレイングマネージャーだ。コードも書くし、朝会の準備もする。ただ正直に言うと、People Managementやメンバーの進捗確認、メンバーをうまく導くといった役割が物凄く苦手だ。得意なのはコードを書くことやインフラを触ること。管理業務は義務としてやっている。

毎朝「昨日何が動いたか」を把握するために、GitHubとSlackを行き来して手動でコンテキストを組み立てていた。これが地味に時間を食う。そこを自動化した。

こういう人に刺さると思う：

- GitHub ProjectとSlackを使っている小規模開発チームのリード
- フルリモート
- 朝の情報収集ルーティンを削りたいエンジニア
- GASでLLM APIを叩いたことはないが試してみたい人

## はじめに

GitHub ProjectとSlackの動きを毎朝まとめてSlack DMに届けるシステムをGASで作った。

途中でGemini APIを試してClaudeに戻すという寄り道をしたのだが、その過程でGASとLLM APIの組み合わせにおける実践的な知見がかなり溜まった。設計判断とハマりどころは後半にまとめる。

---

## 作ったもの

```
7:00〜8:00 JST
  ghp-cache.gs    → GitHub GraphQL API からプロジェクト全件取得 → Google Docs に保存
  slack-cache.gs  → Slack チャンネルの直近24h を取得 → Google Docs に保存

9:00〜10:00 JST
  digest.gs       → 両 Doc を読む
                  → 前日スナップショットと差分計算
                  → Claude API でダイジェスト生成
                  → Slack DM に送信
                  → 蓄積 Doc に追記（NotebookLM 連携用）
```

朝9〜10時にSlack DMで届く。自分から見に行かなくていい。

## なぜCloudflare WorkersやCloud FunctionsではなくGASか

Cloudflare WorkersやCloud Functionsでも同じことはできる。ただこれは会社のプロジェクト管理ツールを個人が勝手に自動化するという、いわば「個人の課題解決」だ。インフラを別途管理したくないし、Google WorkspaceはどうせSlackやGitHubと同じく業務で使っている。GASならGoogle Docsもスクリプトプロパティもトリガーも全部同じ管理画面で完結する。運用コストゼロで動き続けることが最優先だった。

---

## なぜGoogle Docsをキャッシュにするのか — GASの実行時間制限が理由

GASの実行時間制限は6分。GitHub GraphQLのページネーションとSlackのスレッド展開を全部やるとデータ取得だけで数分かかる。そこにLLM APIの呼び出しが加わると6分を超えるリスクがある。

Google Docsを中間キャッシュとして挟むことで**データ取得フェーズと生成フェーズを分離**した。それぞれ独立したGASプロジェクトとして動くため、どちらかが失敗しても影響が局所化される。副産物として「Claude APIが呼ばれた時点のデータ」があとから確認できるのでデバッグが楽になる。

---

## GeminiをGASから呼び出してみた

最初はコスト観点でGemini APIを試した。GeminiにはOpenAI互換エンドポイントがある。GASは`UrlFetchApp`でHTTPリクエストを投げる構造なので、これを使えばコードの変更が少なくて済むと思っていた。

動かすまでに3つの罠にハマった。

1. 認証まわり。`Authorization: Bearer`だけでは動かない。もう一つ必要なヘッダーがある。
2. モデル名。指定したモデルがnew usersには提供終了していた。
3. systemプロンプトの渡し方。OpenAI互換形式でもClaudeとは構造が違う。

3つとも解決したあと、今度は別の問題が出た。

**結局Claudeに戻した。その判断に至った理由と3つの罠の詳細は後半で。**

---

## トークン超過という想定外の壁

Claudeに戻してからも一度詰まった。GitHub Projectの全件データ＋Slackの24h分をそのままAPIに渡すとトークン上限を超える。

解決策の方向性は「渡すデータを削る」だが、どこを削るかの判断が難しい。差分計算に必要なデータと、ダイジェスト生成に必要なデータは別物だからだ。

この設計判断のコードは以下。

```javascript
const cutoff = new Date(Date.now() - 24 * 3600 * 1000);
const recent = (ghpJson.data || []).filter(item =>
  item.updatedAt && new Date(item.updatedAt) >= cutoff
);
```

スナップショット保存は全件のまま、Claude APIに渡す直前だけ絞る。**なぜこの順序でないといけないか**は後半で説明する。

## ClaudeはSlack mrkdwnを知らない

ClaudeにSlack向けのダイジェストを生成させると、最初はMarkdown記法で返してくる。SlackのmrkdwnはMarkdownではないので、そのまま貼るとリンクも太字も崩れる。

基本的な対応表：

| やりがち | 正しい |
|---------|--------|
| `[text](url)` | `<URL\|text>` |
| `**bold**` | `*bold*` |
| `---`（水平線） | 使えない、空行で代替 |

これをsystemプロンプトに書けば9割は解決する。ただ**残り1割に気づくのに時間がかかった**。絵文字や日本語括弧との組み合わせで太字が崩れるエッジケースがある。そのパターンと対処は後半にまとめた。

## NotebookLMとの組み合わせ

毎日のダイジェストを別のGoogle Docsに蓄積している。このGoogle DocsをNotebookLMのソースに追加すると過去のプロジェクト状況を自然言語で検索できる。

使う前に知っておくべき制約が3つある。

- ソースはフォルダ単位で取り込めない、ファイル単位で手動追加
- ソース上限は50件
- Google Driveからインポートしても自動同期されない、手動でre-syncが必要

Google Docsは約100万文字が上限なので、90万文字を超えたらSlack DMで警告を出して新しいGoogle Docsに切り替える実装を入れた。

## やってみてどうだったか

朝イチの行動が変わった。

以前はSlackを開いて昨日の流れを追い、GitHubを開いてチケットの状態を確認して、頭の中でコンテキストを組み立ててから朝会に臨んでいた。この「組み立て」が苦手で、毎朝じわじわと消耗していた。

今はSlack DMを一本読めば終わる。誰が何をやっていて、何が動いて、何が止まっているか。それが届いている。

「あの件どうなったっけ」という過去遡及もNotebookLMに聞けばいい。自分で検索しなくていい。苦手なことを仕組みで補った、という感覚が一番近い。
